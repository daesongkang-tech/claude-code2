## 1. contextの使い方

- 回答・提案・分析を行う前に、必ず該当ロールの `context/` フォルダを参照する
- 新しいインプット（日記・会話・考え事・一次情報）は都度 `context/` に記録する
- ファイル名には必ず日付プレフィックスをつける（例: `2026-03-23_アイデアメモ.md`）
- 情報が古くなった場合は、ファイル冒頭に `[archived]` タグを付与して明示する

---

## 2. アウトプット形式

- 提案・分析は必ず「**結論 → 根拠 → アクション**」の順で出力する
- 数値・KPIは具体的な数字で表現する（「増加する」ではなく「XX%増」）
- 不明点があれば推測で進めず、確認を求めてから進める
- 長文になる場合は見出しを使って構造化する

---

## 3. 役割間の連携ルール

- 他ロールに関わる意思決定を行う場合は、該当ロールのディレクトリも参照する
  - 例：CMOがキャンペーン予算を立案 → CFOの `pl/` や `budget/` を確認する
  - 例：CPOがロードマップを更新 → CSO_salesの `conversations/` を参照して顧客ニーズを反映する
- 単独で判断が難しい場合は「**〇〇（ロール名）と要確認**」と明示してから進める
- ロール間の連携が必要な提案には「**連携先: 〇〇**」を末尾に記載する

---

## 4. 意思決定の基準

- すべての提案に「なぜその選択か」の根拠を添える
- 時間軸を分けて考える：
  - 短期（〜3ヶ月）
  - 中期（3〜12ヶ月）
  - 長期（1年〜）
- トレードオフがある場合は明示して、選択肢を複数提示する

---

## 5. データ管理・更新ルール

- 各ロールのディレクトリ内データは常に最新状態を維持する
- 定期的に `context/` の情報を見直し、古い情報は `[archived]` タグで管理する
- 重要な意思決定・変更は必ずファイルに記録し、口頭・チャットのみで終わらせない

# Claude Code ワークフロールール

## Workflow Orchestration

### 1. Plan Node Default
- 非自明なタスク（3ステップ以上 or 設計判断を含む）は必ずPlan Modeに入ってから着手する
- 途中で想定と外れた場合はすぐにSTOPして再プランニングする。そのまま押し進めない
- 実装だけでなく、検証ステップにもPlan Modeを活用する
- 曖昧さを減らすため、詳細なスペックを最初に書き出す

### 2. Subagent Strategy
- メインのコンテキストウィンドウを汚染しないよう、サブエージェントを積極的に使う
- リサーチ・探索・並列分析はサブエージェントに委譲する
- 複雑な問題ほど、サブエージェントに多くのコンピュートを投入する
- 1サブエージェント = 1タスクに集中させる

### 3. Self-Improvement Loop
- ユーザーから修正を受けたら、必ず `tasks/lessons.md` にそのパターンを記録する
- 同じミスを繰り返さないためのルールを自分自身に書く
- セッション開始時に関連プロジェクトのlessonsを見直す

### 4. Verification Before Done
- 動作を証明せずにタスクを完了済みにしない
- 「スタッフエンジニアがこれを承認するか？」と自問する
- テスト実行・ログ確認・動作実証をしてから完了とする
- mainとの差分を確認し、意図しない変更が入っていないか確かめる

### 5. Demand Elegance（非自明な変更のみ）
- 非自明な変更の場合：「もっとエレガントな方法はないか？」と立ち止まる
- ハック的な修正になっていたら：「今知っていることを前提に、最善の実装をする」
- 単純・明快な修正にはこのステップをスキップする（過剰設計しない）

### 6. Autonomous Bug Fixing
- バグレポートを受けたら、手取り足取り聞かずに自分で調査して修正する
- ログ・エラー・失敗テストを起点に原因を特定して解決する
- ユーザーのコンテキストスイッチを最小化する

## Task Management

1. **Plan First**: `tasks/todo.md` にチェックリスト形式でプランを書く
2. **Verify Plan**: 実装開始前にプランを確認する
3. **Track Progress**: 完了したアイテムを随時マークする
4. **Explain Changes**: 各ステップで変更内容の概要を説明する
5. **Document Results**: `tasks/todo.md` にレビューセクションを追加する
6. **Capture Lessons**: 修正が発生した後に `tasks/lessons.md` を更新する

---

## Core Principles

- **Simplicity First**: 変更はできる限りシンプルに。影響範囲を最小化する
- **No Laziness**: 根本原因を探る。一時しのぎの修正はしない。シニアエンジニア基準で
- **Minimal Impact**: 必要な箇所だけ変更する。新たなバグを持ち込まない
